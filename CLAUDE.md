# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Info Center is a GNOME Shell extension (GNOME 46ÔÇô49) that displays Claude Code API usage in the top panel, and optionally z.ai GLM Coding Plan usage (also in the panel, next to Claude), this month's Redmine time entries per project, and this month's Hubstaff time tracked per project in the dropdown. It is a private rebrand/fork of [Haletran/claude-usage-extension]. Pure GJS (GNOME JavaScript) ÔÇö no build step, no package manager, no test suite.

## Install / reload during development

There is no build. The `update.sh` script is the dev loop: it deletes the installed copy, recopies the repo into `~/.local/share/gnome-shell/extensions/info-center@muzsij`, recompiles the GSettings schema, and then prints the reload instructions for the current session type. It deliberately does **not** restart GNOME Shell itself ÔÇö a detached `gnome-shell --replace` freezes the session on Wayland and can glitch X11 (this is why an earlier version hung the desktop).

```bash
./update.sh       # must be run from the repo root; it `cd ..` then copies the `info-center` dir
```

`./update.sh` only works if the repo directory is named `info-center` (it copies a sibling dir by that literal name). Reloading extension *code* always requires a full shell restart (re-enabling alone doesn't re-import the ESM modules): on **X11** press Alt+F2 ÔåÆ `r`; on **Wayland** there is no in-place reload, so log out and back in.

After editing the schema, you must recompile it or settings reads will fail. Run it in a subshell so the `cd` doesn't leak into the persistent shell's working directory (a leaked `cd schemas` silently breaks the *next* command that uses a repo-root-relative path):
```bash
(cd schemas && glib-compile-schemas .)
```
Or just avoid `cd` entirely: `glib-compile-schemas schemas`. Same rule for any one-off command here ÔÇö prefer absolute/repo-root-relative paths or a subshell over a bare `cd`, because the Bash tool's cwd persists between calls.
`schemas/gschemas.compiled` is gitignored ÔÇö it is a build artifact, never commit it.

## Architecture

Two entry points, loaded by GNOME Shell into different processes. The Shell-side entry point delegates its dropdown sections and fetch logic to four feature modules:

- **`extension.js`** ÔÇö runs in the Shell (compositor) process. `InfoCenterExtension.enable()` creates the panel indicator. `InfoCenterIndicator` (a `PanelMenu.Button`) owns the panel widget (icon; a Claude label + panel progress bar; and a GLM "GLM" tag + label + panel progress bar shown when z.ai is configured), the dropdown menu, the refresh timers, the `Soup.Session`, and the settings signal. It builds nothing inside the dropdown itself beyond the footer separator + **Refresh Now** + **Settings** items; it hands the menu to the four modules to populate.
- **`claudeUsage.js`** ÔÇö the `ClaudeUsage` class. Owns the Claude 5-hour / 7-day dropdown sections and drives the (Claude) panel label + panel progress bar (those panel widgets are created by the indicator and passed into the constructor, because their placement and visibility are governed by the display mode). Reads credentials, fetches usage, renders. This is the only place that touches the Claude credentials/API.
- **`zai.js`** ÔÇö the `ZaiUsage` class. Owns the z.ai GLM Coding Plan 5-hour / weekly dropdown sections and drives its own panel label + panel progress bar (created by the indicator and handed in, like the Claude ones). Off unless a `zai-api-key` is set; when unset it hides both dropdown sections (and their separators) and blanks the panel number. Auth is a static, long-lived API key (no OAuth / token rotation), so it owns just one `Gio.Cancellable` (cancelled-and-replaced per `refresh()`), torn down in `destroy()`. This is the only place that touches the z.ai API. `isConfigured()` is read by the indicator to gate the GLM panel segment's visibility.
- **`usageSection.js`** ÔÇö shared builders for the two-part Claude/GLM usage dropdown sections (title + right-aligned percent, progress bar, reset label), used by both `claudeUsage.js` and `zai.js`: `buildUsageSection(menu, title)` (returns the widgets + the menu `item` for visibility toggling), `applyBarWidth(bar, bg)` / `updateProgressBar(bar, bg, usage)` (the fill width against the bg's allocated width + the `usage-*` color class at 40/70/90), `updatePanelProgressBar(panelBar, usage)` (the 0ÔÇô50px panel bar), and `formatResetCountdown(when)` (a Date/ISO string/ms epoch ÔåÆ `"1d 3h"` / `"2h 5m"` / `"5m"` / `"now"` / `"ÔÇö"`).
- **`redmine.js`** ÔÇö the `Redmine` class. Owns the Redmine dropdown sections (Today / Tomorrow issue lists + this-month per-project time totals), fetches issues and time entries, and renders them. This is the only place that touches Redmine. It owns two resources ÔÇö a `Gio.Cancellable` threaded into its fetches and a shared earnings `Tooltip` (see `tooltip.js`) ÔÇö both torn down in `destroy()`.
- **`hubstaff.js`** ÔÇö the `Hubstaff` class. Owns the single "Hubstaff time ÔÇö this month" dropdown section (one row per project you tracked time on this month, summed, most-tracked first), handles the OAuth token exchange, fetches organizations / projects / daily activities, and renders them. This is the only place that touches Hubstaff. It owns two `Gio.Cancellable`s ÔÇö one threaded into its data fetches (cancelled-and-replaced per `refresh()`), and a separate `_tokenCancellable` for the token exchange (so an overlapping refresh can't abort a refresh-token rotation mid-flight) ÔÇö plus a shared earnings `Tooltip`, all torn down in `destroy()`.
- **`sections.js`** ÔÇö shared builders for the Redmine/Hubstaff monthly-totals dropdown sections, which have the same shape: `buildTotalsSection(menu, title)` (separator + title row with right-aligned total + rows box, hidden initially), `addTotalsRow(rowsBox, name, value)` (one name/H:MM row, returned so the caller can bind an earnings tooltip), `messageLabel(text)` (dim error/empty-state label), and `formatHM(hours)` (fractional hours ÔåÆ `H:MM`; Hubstaff passes `seconds / 3600`).
- **`tooltip.js`** ÔÇö the shared `Tooltip` class and a `formatMoney(amount, currency)` helper, used by both `redmine.js` and `hubstaff.js` for the earnings tooltip. A `Tooltip` owns one floating `St.Label` (`info-center-tooltip`) created lazily in `Main.layoutManager.uiGroup` (so it can paint above the menu's boxpointer, **not** under `this.menu`). `bind(actor, getText)` makes a row reactive + `track_hover` and, on hover, reads `getText()` (returning `''` suppresses the tooltip), positions the label at the pointer clamped to the monitor, and raises it above its siblings; `hide()` hides it on re-render; `destroy()` reaps it. It is the only place that touches `Main`/the uiGroup for tooltips.
- **`prefs.js`** ÔÇö runs in a separate preferences process (GTK4/Adwaita, not Shell). It only reads/writes GSettings keys; it shares no runtime state with `extension.js`. The two communicate exclusively through GSettings.

All four feature modules receive the `Soup.Session` through a `getSession` getter (`() => this._session`) rather than a stored reference, so when the indicator recreates the session on a proxy change the modules pick up the new session on their next request. They are passed `settings` directly and read their own keys. `Redmine` and `Hubstaff` create no timers or signals of their own, but each owns a `Gio.Cancellable` for its in-flight fetches (Hubstaff owns a second one, `_tokenCancellable`, for its token exchange) and exposes `destroy()` to cancel it. `ZaiUsage` likewise owns a single `Gio.Cancellable` (and the menu open-state signal + a deferred bar-reapply idle) torn down in `destroy()`. `ClaudeUsage` owns a short-lived auth-retry timer and a `Gio.Cancellable` for its in-flight async I/O (see below) and exposes `destroy()` to tear both down; the indicator's `destroy()` calls `this._claude.destroy()`, `this._zai.destroy()`, `this._redmine.destroy()`, and `this._hubstaff.destroy()`. All menu items are owned by `this.menu` (destroyed by `super.destroy()`).

The indicator drives four independent refresh timers (`refresh-interval` for Claude, `zai-refresh-interval` for GLM, `redmine-refresh-interval` for Redmine, `hubstaff-refresh-interval` for Hubstaff) through a shared `_startTimer` / `_stopTimer` / `_restartTimer` helper over the `this._timers` table (keyed `claude` / `zai` / `redmine` / `hubstaff`, each entry holding its interval key and module), all stopped in `destroy()`. `_refreshNow()` (the initial refresh, the **Refresh Now** menu item, and nothing else) refreshes every module and restarts its timer so the next tick lands a full interval later. In the extension class, `panel-box` / `panel-position` changes are debounced (`_schedulePlaceIndicator`, 400 ms) before re-creating the indicator, since a re-place destroys and rebuilds everything (four fetches) and the position SpinRow writes once per click.

The GSettings schema (`schemas/*.gschema.xml`) is the contract between Shell and prefs. Adding a setting means: add the key to the schema XML, add a UI row in `prefs.js`, recompile the schema, and (if it affects the panel live) handle it in the `settings.connect('changed', ...)` switch in `extension.js._init` (which dispatches `zai-api-key` to `this._zai.refresh()` + `_updateDisplayMode()` so the GLM panel segment appears/disappears with the key; `redmine-*` keys to `this._redmine.refresh()` ÔÇö except `redmine-hourly-rate` / `redmine-currency`, which go to `this._redmine.rerender()` ÔÇö `hubstaff-personal-access-token` to `this._hubstaff.refresh()`, and `hubstaff-hourly-rate` / `hubstaff-currency` to `this._hubstaff.rerender()`).

### Claude usage data flow (claudeUsage.js)

1. The indicator's Claude timer fires every `refresh-interval` seconds (`_startClaudeTimer`) and calls `this._claude.refresh()` (also called on `_init` and after a proxy-driven session recreation). Redmine has its own independent timer (`redmine-refresh-interval` / `_startRedmineTimer`) so the two refresh cadences are configured separately.
2. `ClaudeUsage.refresh()` reads the OAuth token from `$CLAUDE_CONFIG_DIR/.credentials.json` (falls back to `~/.claude/.credentials.json`), at `json.claudeAiOauth.accessToken`.
3. `_fetchUsage(token)` GETs `https://api.anthropic.com/api/oauth/usage` with `Authorization: Bearer <token>` and the `anthropic-beta: oauth-2025-04-20` header.
4. `_updateDisplay(data)` renders `data.five_hour.utilization` and `data.seven_day.utilization` (percentages) into the panel label, panel progress bar, and dropdown.

Stale-token handling: right after login the on-disk token is often expired (Claude Code refreshes it lazily). `refresh()` skips the request when `claudeAiOauth.expiresAt` is already past, and `_fetchUsage` treats an HTTP 401 the same way ÔÇö both route into `_handleStaleToken`, which shows a soft "RefreshingÔÇª" state (keeping the last good percentages if `_hasData`, but flagging the reset labels so frozen numbers aren't presented as current) and arms a one-shot `_scheduleRetry` timer (`AUTH_RETRY_SECONDS`) that re-reads the credentials file, so a refreshed token is picked up in ~20s instead of waiting a full `refresh-interval`. Both stale paths share one `_authRetries` budget: `_handleStaleToken` counts one retry **per scheduled round** (it no-ops when a retry timer is already pending, so the main timer and retry timer firing together don't burn the budget per-tick) and shows a hard `Error` after `MAX_AUTH_RETRIES`; the counter resets on a successful fetch. `ClaudeUsage` owns two resources, both torn down in `destroy()`: the retry timer (`_clearRetry`) and a `Gio.Cancellable` (`_cancellable`) threaded into the credential read and the usage fetch ÔÇö `refresh()` cancels the previous one before starting, so overlapping refreshes don't race and callbacks can't `set_text` on widgets `super.destroy()` has disposed.

Usage thresholds drive the `usage-low/medium/high/critical` CSS classes (40/70/90) in `updateProgressBar` (in `usageSection.js`, shared with GLM) ÔÇö these class names are defined in `stylesheet.css`.

### GLM (z.ai) usage data flow (zai.js)

Optional, off by default. Configured with a single `zai-api-key` (created at z.ai/manage-apikey/apikey-list) plus `zai-refresh-interval`, set from the **GLM** prefs page.

1. `ZaiUsage.refresh()` runs on `_init`, on every tick of its own `zai-refresh-interval` timer, whenever `zai-api-key` changes, and after a proxy-driven session recreation. With no key it blanks the panel number, hides both dropdown sections, and bails; otherwise it un-hides them and fetches.
2. `_fetchUsage(token)` GETs `https://api.z.ai/api/monitor/usage/quota/limit` with `Authorization: <token>` (**no** `Bearer` prefix, unlike Claude/Hubstaff), `Accept-Language`, and `Content-Type: application/json`.
3. The response's `data.limits[]` array holds token-limit entries keyed by `(unit, number)`: the **5-hour** window is `(unit=3, number=5)` and the **weekly** window is `(unit=6, number=1)`. `_updateDisplay` picks those two out (`_findLimit`), reads each entry's `percentage` (0ÔÇô100) and `nextResetTime` (Unix **ms**), and renders them into the panel number (5-hour), panel progress bar, and the two dropdown sections. A 401/403 shows an "Auth error / Check API key" state.

The GLM panel segment (a dim "GLM" tag + label + panel progress bar) sits to the right of the Claude number. Its text/bar/both visibility is driven by its **own** `zai-display-mode` key (independent of Claude's `display-mode`) in `_updateDisplayMode`, gated on `this._zai.isConfigured()` so it appears only when a key is set. The `.info-center-zai-prefix` tag disambiguates the two panel percentages.

### Redmine data flow (redmine.js)

Optional, off by default. Configured via the `redmine-url`, `redmine-api-key`, `redmine-projects`, `redmine-project-names`, `redmine-statuses`, `redmine-tasks-all-projects`, and the **Earnings** group (`redmine-hourly-rate` + `redmine-currency`) GSettings keys (set from the **Redmine** prefs page).

1. `Redmine.refresh()` runs on `_init`, on every tick of its own `redmine-refresh-interval` timer, and whenever a `redmine-*` setting changes. It hides every section and bails unless URL + API key are set and either at least one project is selected or all-projects task mode is on.
2. `_fetchIssues()` GETs `<redmine-url>/issues.json?assigned_to_id=me&status_id=*` (paged, `limit=100`), then `_updateIssuesDisplay()` filters by selected projects (unless `redmine-tasks-all-projects`) and statuses (`redmine-statuses`; empty = all), and buckets each issue into the **Today** / **Tomorrow** sections by its `start_date`..`due_date` span. Rows are clickable buttons that open `<redmine-url>/issues/<id>`. `_setIssuesMessage` shows errors there.
3. `_fetchTimeEntries()` GETs `<redmine-url>/time_entries.json?user_id=me&from=<1st of month>&to=<today>` (paged, `limit=100`), summing `hours` per `project.id`. It only runs when at least one project is selected (monthly totals are project-scoped). `_updateDisplay()` renders one row per selected project (name + `H:MM`-formatted hours via `formatHM` from `sections.js`), falling back to `redmine-project-names` then `Project #<id>` for labels. `_setMessage` shows errors in that spot.

**Earnings (tooltip).** Like Hubstaff, the monthly-totals section shows estimated earnings in a tooltip on hover, derived at render time from the **manual** `redmine-hourly-rate` (├ù currency label `redmine-currency`) ÔÇö `_updateDisplay` caches the raw fetch (`this._hoursByProject` / `this._namesByProject`) and computes `hours ├ù rate` per project (no `/3600` ÔÇö Redmine totals are already in hours). It sets a per-project "Earned: ÔÇª" tooltip and a month total in `this._totalTooltip`, both only when earnings > 0 (an empty `_totalTooltip` disables the title-row tooltip), and binds them via the shared `Tooltip` (`tooltip.js`). A rate/currency change needs no refetch: `extension.js` watches `redmine-hourly-rate` / `redmine-currency` and calls `Redmine.rerender()` (re-runs `_updateDisplay` from the cache; no-op until the first fetch). `_setMessage` and `_hideTotals` clear the cache + `_totalTooltip` and hide the tooltip, so a later rate change can't re-render stale data over an error or resurrect a section hidden because no project is selected.

All Redmine requests use the `X-Redmine-API-Key` header and reuse the indicator's `Soup.Session` (via the getter) and settings signal, driven by the Redmine refresh timer (`redmine-refresh-interval`). They are threaded through a single `Gio.Cancellable` that `refresh()` cancels-and-replaces on each call, so overlapping refreshes (a timer tick racing a `redmine-*` settings change) don't interleave on the display and post-teardown callbacks can't touch destroyed widgets; the indicator's `destroy()` calls `this._redmine.destroy()` to cancel it. The indicator debounces `redmine-*` settings changes (`_scheduleRedmineRefresh`, 400 ms) so ticking several project/status checkboxes in prefs triggers one fetch rather than one per click.

The **prefs.js** Redmine page (`_buildRedminePage`) is the only place that fetches the project list: "Fetch projects" (and auto-fetch on tab `map`) calls `_fetchProjects`, which GETs `<redmine-url>/projects.json`, renders a checkbox per project into `redmine-projects`, and caches idÔåÆname into `redmine-project-names` so the panel can label projects before its own fetch returns.

### Hubstaff data flow (hubstaff.js)

Optional, off by default. Configured entirely from the **Hubstaff** prefs page (`_buildHubstaffPage`): a refresh interval, a single Personal Access Token, and an **Earnings** group (`hubstaff-hourly-rate` + `hubstaff-currency`). There is no project selection ÔÇö the section shows every project you tracked time on this month, summed.

**Auth (token rotation).** A Hubstaff Personal Access Token (PAT) is itself a long-lived (90-day) OAuth *refresh* token, created at developer.hubstaff.com with the **`hubstaff:read`** scope. There are four token-related GSettings keys:
- `hubstaff-personal-access-token` ÔÇö the seed the user pastes in prefs. **The user-entered value; the extension never overwrites it.** This is the only token key watched in `extension.js` (its change triggers `this._hubstaff.refresh()`).
- `hubstaff-refresh-token` ÔÇö the rotating refresh token. `_ensureAccessToken` checks the cache, then delegates to `_exchangeRefreshToken`, which exchanges the current refresh token at `POST https://account.hubstaff.com/access_tokens` (`grant_type=refresh_token`, `application/x-www-form-urlencoded`, **no** client id/secret for a PAT); the response carries a *new* refresh token that **replaces** the old one (Hubstaff rotates on every exchange ÔÇö the old one is dead the moment the request reaches Hubstaff), written back here. `_currentRefreshToken()` prefers this, falling back to the seed. Because losing the rotated token in the response permanently bricks auth, the exchange runs under its **own** `Gio.Cancellable` (`_tokenCancellable`) that a normal `refresh()` never cancels (only `destroy()` does), and concurrent callers queue onto the single in-flight exchange via `_tokenWaiters` rather than each starting their own with the same about-to-die refresh token. For the same reason the indicator's `_recreateSession()` deliberately **does not** `session.abort()` the old session (which would cancel the in-flight exchange too) ÔÇö it lets the old session drain and relies on each module's own cancellable to drop its stale data fetch. When the exchange fails (non-200, unexpected body, or transport error) `_exchangeRefreshToken` serves every queued waiter `null` via `_drainTokenWaiters` (callers treat `null` as "abort this round") so a failed exchange can't leave a fetch chain hanging; on success it serves the access token. If the token response omits `expires_in`, the cache expiry falls back to `DEFAULT_ACCESS_TOKEN_TTL` (24h) so a missing field can't mark the cache instantly stale and force a re-exchange (and refresh-token rotation) on every refresh.
- `hubstaff-access-token` / `hubstaff-token-expires-at` ÔÇö the cached short-lived (24ÔÇô72h) access token and its Unix expiry. The cache is reused until within 60s of expiry, so the token endpoint is hit rarely (roughly once a day), not on every refresh. If a request is rejected with **HTTP 401** *before* that cached expiry (server-side revocation or clock skew), `_apiGet` expires the cache (`hubstaff-token-expires-at` ÔåÉ 0), re-exchanges the still-valid refresh token once, and replays the request with the fresh token ÔÇö so the section self-recovers without the user re-entering a PAT (a `retried` flag bounds it to one retry).

None of the three extension-written keys (`hubstaff-refresh-token`, `hubstaff-access-token`, `hubstaff-token-expires-at`) are watched in `extension.js`, so token rotation does not retrigger a refresh. When the user enters a new PAT, `_buildHubstaffPage` clears all three first, then writes the seed last, so the next refresh exchanges with the new token. Re-applying the **same** PAT writes a byte-identical seed, which dconf suppresses (no `changed` signal, no refresh) ÔÇö so the apply handler first clears the seed to `''` when the value is unchanged, forcing a value transition so the final write always re-triggers the refresh (lets a user recover from an auth error by re-pasting the same token).

**Fetch.** With a valid access token (`Authorization: Bearer <token>`), `refresh()` chains: `GET /v2/users/me` (own user id) ÔåÆ `GET /v2/organizations` (paged via `pagination.next_page_start_id` / `page_start_id`) ÔåÆ for each org sequentially, `GET /v2/organizations/<id>/projects` (idÔåÆname) then `GET /v2/organizations/<id>/activities/daily?date[start]=<1st of month>&date[stop]=<today>&user_ids=<me>` (the bracketed keys are percent-encoded; `user_ids` scopes totals to the signed-in user even with manager/owner visibility). It sums `tracked` (seconds) per `project_id` across all orgs, then `_updateDisplay` renders one `H:MM` row per project (most-tracked first), labelling unknown ids `Project #<id>`. `_setMessage` shows token / fetch errors in that spot. The data fetches are threaded through one `Gio.Cancellable` that `refresh()` cancels-and-replaces, so overlapping refreshes don't interleave and post-teardown callbacks can't touch destroyed widgets; the token exchange uses the separate `_tokenCancellable` (see Auth above) so an overlapping refresh can't abort it mid-rotation. `destroy()` cancels both.

**Earnings (tooltip).** Earnings are estimated from the user's **manual** hourly rate (`hubstaff-hourly-rate`) ÔÇö Hubstaff's Rates API needs manager/owner access that a personal token lacks (it 403s), so it is not used. Earnings are **derived at render time, not fetched**: `_updateDisplay` caches the raw fetch (`this._secondsByProject` / `this._namesByProject`), reads the current rate + currency (`hubstaff-currency`), and computes `tracked/3600 ├ù rate` per project. It builds tooltip strings (per-project "Earned: ÔÇª" and a month total in `this._totalTooltip`, both only when earnings > 0 ÔÇö an empty `_totalTooltip` disables the title-row tooltip) and binds them via the shared `Tooltip` (`tooltip.js`), whose `bind` makes a row reactive + `track_hover` so hovering shows the floating `info-center-tooltip` label above the menu. Because earnings are render-derived, a rate/currency change needs no refetch: `extension.js` watches `hubstaff-hourly-rate` / `hubstaff-currency` and calls `Hubstaff.rerender()`, which re-runs `_updateDisplay` from the cache (no-op until the first fetch). `_setMessage` clears the cache so a later rate change can't re-render stale data over an error.

### Things that must stay consistent

- **Three display modes** (`text`/`bar`/`both`) are handled in `_updateDisplayMode` (in `extension.js`, since they govern the panel widgets the indicator owns), defined in the schema, and selected in `prefs.js`. The indexÔåöstring mapping (`['text','bar','both']`) is duplicated across `extension.js` and the two `prefs.js` display-mode rows ÔÇö keep them in sync. Claude and GLM have **separate** display modes: the Claude widgets follow `display-mode`, the GLM widgets follow `zai-display-mode` (so GLM can be, e.g., bar-only while Claude is both). `_updateDisplayMode` reads both keys and applies each; the GLM ones (tag + label + bar) are additionally gated on `this._zai.isConfigured()`, so they only appear when a z.ai key is set. Both keys route to `_updateDisplayMode` in the settings switch.
- **Lifecycle / leaks:** the indicator's `destroy()` must stop all four refresh timers (Claude + GLM + Redmine + Hubstaff), remove the debounce timer, call `this._claude.destroy()`, `this._zai.destroy()`, `this._redmine.destroy()`, and `this._hubstaff.destroy()`, abort the Soup session, and disconnect the settings signal. All four feature modules own a `Gio.Cancellable` (and `ClaudeUsage` a retry timer; `ZaiUsage` a menu open-state signal + a bar-reapply idle) torn down in their `destroy()` ÔÇö if you add another timer, signal, or session inside `claudeUsage.js` / `zai.js` / `redmine.js` / `hubstaff.js`, extend that teardown and ensure the indicator calls it, or it leaks across enable/disable cycles (a common reason extensions get rejected from extensions.gnome.org). The earnings `Tooltip` (`tooltip.js`) owned by both `Redmine` and `Hubstaff` is an extra wrinkle: its `St.Label` lives in `Main.layoutManager.uiGroup`, **not** under `this.menu`, so `super.destroy()` won't reap it ÔÇö each module's `destroy()` calls `this._tooltip.destroy()`.
- **`metadata.json` `version`** is an integer that must be bumped on each release submitted to extensions.gnome.org. `shell-version` lists supported GNOME majors.

## GNOME conventions to follow here

- Async I/O only ÔÇö use `*_async`/`*_finish` (as `ClaudeUsage.refresh`/`_fetchUsage` do). Never block the Shell's main loop with sync file or network calls.
- Imports use `gi://` (GObject introspection) and `resource:///org/gnome/shell/...` (Shell internals). These resolve only inside the Shell runtime, not under plain `node`.
- The icon is loaded from disk via `GLib.build_filenamev([extensionPath, ...])`; `extensionPath` comes from `this.path` and is passed into the indicator ÔÇö don't hardcode paths.
